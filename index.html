Background

Telstra is looking to expend their current capability to deploy configurations by including Ansible, AWX and Gitlab into the automation framework.

To do this a Proof of concept is being built in the Telsta BTS - NAB - Lab.

To ensure integration capability a number of supporting services need to be built, this will include:

  • Solarinds

  • Infoblox

  • Storage

  • Windows LADP Authentication

  • TACACS+

  • Jenkins

Referance documentation

  • Archetectural Referance - NAB Ansible - POC

  • Ansible POC Teams Site

  • Ansible POC Sharepoint

Build

The build is split into phases to allow testing to begin earlier in the project. Additional build phases can be running in parallel with testing in the phase 1 environment.

LTM & GTM Endpoints - Existing LAB Viprion Chassis will have a new vlan trunked to support intergration into the Ansible POC.

Switching Endpoints - Dedicated switches will be deployed for the POC.

Interconnect Switches - Shared L2 and L3 switches will be used for the network layer connectivity for the POC environment. Dedicated VLANS and IP addressing have been created to provide logical network isolation.

Device Naming

Device naming standard is based the standard used by the NAB BAU support team in production server deployments. This standard is supported as an option in the current standards document Naming Conventions for IP Devices although it doesn’t specify the actual formatting.

The BAU engineering team provided me the following formatting to follow:

sa3[knx|dpk]<nn><xx>
    <nn> numeric server count
    <xx> defines function

Splunk

Purpose

Asible integration with splunk is used to provide visibility and reporting on job status and statistics.

Splunk Architecture

Splunk has two primary components:

  • Data Collection Node (DCN)

  • Splunk Scheduler (includes Distributed collection Scheduler)

Splunk Resource requirements

Each Data Collection Node (DCN) needs at least one CPU core for every 10 hosts from which the DCN is collecting data. Recommended maximum of 40 host and 1000 nodes per DCN.

Based on these recommendations we can estimate that 250 nodes per CPU is sufficient. In our we can use the minimum recommended spec as our automation environment will not go over 200 nodes sending to Splunk.

Further details can be found in the Splunk deployment guide

A production deployment of splunk if receiving logs from all devices in the network will easily go over this minimum spec.

Test Cases

Testing approach

Where possible test cases will be written as playbooks. This will provide a number of benefits to the project long term:

  • Ability to re-run test cases for minimal effort due to design changes.

  • Ability to re-use test cases in production deployment.

  • Ability to scale up the number of tests being run concurrently for capacity testing.

  • Ability to re-run test cases in failure scenarios. This will be a very significant time saving.

  • Ability to use doc-as-code approach for automatic documentation of test cases.

Testing Phases

Testing will be broken up into 3 phases:

  1. Phase 1 - Testing to validate functionality and interoperability

  2. Phase 2 - Testing to validate HA fail-over both local site (where applicable) and cross-site.

  3. Phase 3 - Load testing of the infrastructure to determine hardware requirements for production build.

During phase 1 test cases will be used to prove functionality. Once phase 2 is complete, we will use the same test cases to validate each failure scenario.
